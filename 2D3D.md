# ADR: Separate 2D and 3D Render Passes

## Context

`Renderer::RenderFrame()` currently interleaves 3D (perspective projection, depth-tested)
and 2D (orthographic projection, depth-off overlay) drawing.  Several subsystems
independently switch between projections mid-frame:

| Subsystem | Current behaviour |
|---|---|
| `Location::Render` / `GlobalWorld::Render` | Pure 3D |
| `ExplosionManager::Render` | Pure 3D |
| `ParticleSystem::Render` | Pure 3D |
| `UserInput::Render` | Calls `BeginText2D`/`EndText2D` internally (2D) |
| `GameCursor::Render` | Mixes 3D markers with 2D cursors; calls `SetupMatricesFor2D`/`SetupMatricesFor3D` up to 6 times |
| `TaskManagerInterface::Render` | Calls `RenderTargetAreas` (3D particles) then `BeginText2D` for all UI (2D) |
| `StartSequence::Render` | MIXED: 2D captions with own `OrthoOffCenterRH`, then `SetupMatricesFor3D` + 3D grid lines |
| HUD block in `RenderFrame` | Pure 2D inside `g_editorFont.BeginText2D`/`EndText2D` |

Each 2D↔3D switch changes the projection matrix and associated GL state, which under the
D3D12 translation layer means different `DrawConstants` uploads and potentially different
PSO keys (depth enable/disable, blend mode).  These round-trips are unnecessary and
fragment the upload ring buffer.

## Decision

Split `RenderFrame()` into two ordered passes:

1. **3D pass** — perspective projection, depth test/write enabled.
2. **2D pass** — orthographic projection, depth test disabled, depth write disabled.

Subsystems that currently mix both will expose separate `Render3D()` / `Render2D()` methods.

`GameCursor` is split into two classes — `GameCursor` (coordinator + 3D) and
`GameCursor2D` (screen-space) — so that the 2D class can later adopt `SpriteBatch`
without touching the 3D legacy path.  See **Future Work** at the end of this document
for the separate SpriteBatch migration plan.

## Subsystem Changes

### 1. `GameCursor` / `GameCursor2D` — class split

`GameCursor::Render()` (gamecursor.cpp:274-515) is the most tangled subsystem.  It
mixes 3D world-space cursors (placement markers, move-here arrows, highlight rings on
terrain) with 2D screen-space cursors (standard arrow, selection halos projected to screen
coords, turret crosshair).

The 2D and 3D paths use fundamentally different vertex formats (`glVertex2f` vs
`glVertex3fv`), coordinate systems, and GL state.  When `SpriteBatch` replaces the legacy
2D immediate-mode calls, the 2D path changes completely while the 3D path is untouched.
A class boundary at this seam means `SpriteBatch` adoption touches only `GameCursor2D`,
with zero changes to `GameCursor`.

Both classes share the same per-frame state.  `PrepareCursorState()` runs once before
either pass, filling a struct that both read:

```cpp
struct CursorFrameState
{
  // Input state
  int screenX, screenY;
  LegacyVector3 mousePos;

  // Gameplay queries (filled by PrepareCursorState)
  bool somethingSelected;
  bool somethingHighlighted;
  WorldObjectId selectedId, highlightedId;
  LegacyVector3 selectedWorldPos, highlightedWorldPos;
  float highlightedRadius;
  bool isEditing;
  bool isInteractive;
  bool inLocation;
  bool inGlobalWorld;             // locationId == -1
  int taskType;                   // e.g. GlobalResearch::TypeRadarDish
  Task* currentTask;

  // Pre-computed screen projections (computed during 3D pass while
  // perspective matrices are active — Get2DScreenPos reads the current
  // GL projection/modelview stacks, so it must run before the 2D pass
  // overwrites them with ortho).
  float selectedScreenX, selectedScreenY;
  bool selectedOnScreen;
  float highlightedScreenX, highlightedScreenY;
  bool highlightedOnScreen;
  // Off-screen edge data for FindScreenEdge
  float selectedEdgeX, selectedEdgeY;
  Vector2 selectedEdgeNormal;
  float highlightedEdgeX, highlightedEdgeY;
  Vector2 highlightedEdgeNormal;
  // Radar dish entrance (only valid when radarDishEntranceValid == true).
  // This is a third world position (from dish->GetEntrance()) that is neither
  // selectedWorldPos nor highlightedWorldPos.  Must be projected via
  // Get2DScreenPos during the 3D pass.
  bool radarDishEntranceValid;
  float radarDishEntranceScreenX, radarDishEntranceScreenY;
  // Pre-computed camera distance for highlighted-entity halo sizing.
  // Pure math (no matrix dependency), but pre-computed here to keep
  // GameCursor2D::Render as a pure draw call with no game-object lookups.
  // Used as: highlightedRadius * 100 / sqrt(highlightedCamDist).
  float highlightedCamDist;       // (cameraPos - highlightedWorldPos).Mag()
  // Selection arrow alive checks (moved from RenderSelectionArrows draw path
  // to avoid gameplay queries — GetEntity / GetUnit / m_dead / NumAliveEntities
  // — inside GameCursor2D::Render).
  bool selectionArrowsValid;      // entity/unit alive and drawable

  // Selection arrow state (written by GameCursor, read + updated by GameCursor2D)
  float selectionArrowBoost;      // copied from m_selectionArrowBoost

  // Cross-pass flags
  bool suppressStandardCursor;    // set by PrepareCursorState for mode-based
                                  // suppressions (cut scene, entity track) and
                                  // by Render3D / GameCursor2D::Render when a
                                  // specific cursor is drawn.  Suppresses the
                                  // fallback standard cursor at the end of
                                  // GameCursor2D::Render.
  bool cursorRendered3D;          // set by Render3D when a 3D cursor is actually drawn
};
```

`GameCursor2D` accesses the frame state via `m_parent.GetFrameState()`.  This avoids
re-querying selection state and makes both render methods pure draw calls with no
gameplay queries.

#### Critical: `Get2DScreenPos` requires 3D matrices

`RenderSelectionArrows` (gamecursor.cpp:673-750) calls `Camera::Get2DScreenPos()` to
project world positions onto the screen.  `Get2DScreenPos` reads the **current GL
projection and modelview stacks** via `gluProject` — it is not a pure camera method.
If `RenderSelectionArrows` runs in the 2D pass, the active matrices are orthographic
and all projections will be wrong.

**Solution**: `PrepareCursorState()` (which runs in the 3D pass while perspective
matrices are active) pre-computes screen positions for all objects that need selection
arrows.  It calls `Get2DScreenPos`, `FindScreenEdge`, and stores results in
`CursorFrameState`.  `GameCursor2D::RenderSelectionArrows` reads pre-computed values
only — no `Get2DScreenPos` call, no matrix dependency.

`FindScreenEdge` (gamecursor.cpp:564-626) is pure screen-space math — it
intersects a line direction with screen edges using `ScreenH()`/`ScreenW()` and
does **not** call `Get2DScreenPos` itself.  However, its **caller** in the
off-screen arrow path (line 734) calls `Get2DScreenPos` to compute the line
direction before passing it to `FindScreenEdge`.  The entire off-screen arrow
computation — including the `Get2DScreenPos` call, `lineNormal` calculation,
and `FindScreenEdge` call — must therefore run in `PrepareCursorState`, not in
`GameCursor2D::Render`.

#### `BoostSelectionArrows` external callers

`BoostSelectionArrows(float seconds)` is called from outside `GameCursor`:
- `taskmanager.cpp:601` — `g_app->m_gameCursor->BoostSelectionArrows(2.0f);`
- `team.cpp:111` — `g_app->m_gameCursor->BoostSelectionArrows(2.0f);`

The method modifies `m_selectionArrowBoost`, which `RenderSelectionArrows` reads and
decrements each frame.  After the split:
- `m_selectionArrowBoost` stays on `GameCursor` (preserves external API).
- `PrepareCursorState()` copies it into `CursorFrameState::selectionArrowBoost`.
- `GameCursor2D::RenderSelectionArrows` reads the frame-state copy and computes the
  decremented value.
- After `GameCursor2D::Render()`, the decremented value is written back via
  `GameCursor::WriteBackArrowBoost()`.  (Alternative: move the decrement into
  `PrepareCursorState` and make `RenderSelectionArrows` read-only.)

#### `RenderSelectionArrow` GL state corruption

`RenderSelectionArrow` (gamecursor.cpp:629-670) sets up and restores GL state
internally with **hardcoded enables/disables** — the same class of bug documented
for `UserInput::Render` in §3:

```cpp
// gamecursor.cpp:638-640  — setup (matches 2D contract ✓)
glEnable    ( GL_BLEND );
glDisable   ( GL_CULL_FACE );
glDepthMask ( false );

// ... draw quad ...

// gamecursor.cpp:666-668  — restore (BREAKS 2D contract ✗)
glDepthMask ( true );              // ← 2D contract: false
glDisable   ( GL_TEXTURE_2D );     //   OK (per-draw-call state)
glEnable    ( GL_CULL_FACE );      // ← 2D contract: disabled
```

After the last `RenderSelectionArrow` call, depth writes are re-enabled and cull
face is on.  Every subsequent 2D draw (`TaskManagerInterface::Render2D`, standard
cursor, HUD) will render with corrupted state.

**Solution**: When moving `RenderSelectionArrow` to `GameCursor2D` in Step 4, strip
the pass-level state setup/restore (lines 638-640 and 666-668).  The outer 2D pass
owns the state contract.  Only per-draw-call state (`GL_TEXTURE_2D` enable,
`glBlendFunc` changes) should remain.

`AdviseHighlightingSomething()`, `AdvisePlacementOpportunity()`, and
`AdviseMoveableEntitySelected()` are public accessors with **zero external callers**
(confirmed via workspace-wide grep).  The backing flags (`m_highlightingSomething`,
`m_validPlacementOpportunity`, `m_moveableEntitySelected`) are set in `Render()` but
never read outside `gamecursor.cpp`.  Remove them during the split to shrink the API.

`RenderWeaponMarker()` is public but has no external callers (only called from within
`GameCursor::Render()` at line 396).  Make it private.

`m_cursorMissile` is unused — remove it (already noted in the resource table).

The entity track mode branch (gamecursor.cpp:376-401) wraps `RenderWeaponMarker`
in `if (false) { ... }`, making the entire block just `cursorRendered = true` (i.e.
a mode-based cursor suppression with no actual drawing).  Remove the `if(false)`
body during the split.  The `cursorRendered = true` becomes a
`suppressStandardCursor = true` assignment in `PrepareCursorState`.

#### Why a class split, not just methods

If `Render3D()` and `Render2D()` were methods on the same class, the future SpriteBatch
migration would require modifying `GameCursor` — a file that also contains 3D legacy
code, `MouseCursor` resource construction, and gameplay query logic.  Every change to
the 2D rendering path risks regressions in the 3D path.

With `GameCursor2D` as a separate class:
- SpriteBatch adoption is scoped to `gamecursor_2d.cpp` — no edits to `gamecursor.cpp`.
- `GameCursor2D` can hold a `SpriteBatch&` member without pulling SpriteBatch headers
  into the 3D code path.
- `GameCursor2D` can be independently unit-tested with a mock frame state.
- The `GameCursor` constructor shrinks (no longer creates 2D-only resources).
- File ownership is clear: 3D cursor bugs → `gamecursor.cpp`, 2D cursor bugs →
  `gamecursor_2d.cpp`.

#### Files changed

- `gamecursor.h` — Add `CursorFrameState`, `PrepareCursorState()`, `Render3D()`,
  `GetFrameState()`, `WriteBackArrowBoost()`.  Remove 2D-only `MouseCursor*` members
  and `RenderSelectionArrow`/`FindScreenEdge`.  Remove dead `Advise*()` accessors and
  their backing flags.  Make `RenderWeaponMarker` private.  Remove `m_cursorMissile`.
  Keep legacy `Render()` wrapper during transition.
- `gamecursor.cpp` — Move 3D draw code into `Render3D()`, move `PrepareCursorState()` out.
  `PrepareCursorState` calls `Get2DScreenPos` and `FindScreenEdge` for all objects that
  need selection arrows (must run while perspective matrices are active).  Add profiler
  scopes: `"Prepare GameCursor"` and `"Render GameCursor 3D"`.  Remove 2D draw code

- `gamecursor_2d.h` (NEW) — `GameCursor2D` class declaration.
- `gamecursor_2d.cpp` (NEW) — All 2D draw code, selection arrows (read pre-computed

### 2. `TaskManagerInterfaceIcons` — moderate complexity

`TaskManagerInterfaceIcons::Render()` (taskmanager_interface_icons.cpp:340-425) has one
3D call (`RenderTargetAreas`, line 351) before switching to 2D for everything else.

#### Proposed split

```
TaskManagerInterfaceIcons::Render3D()
├─ Early-out checks (editing, no location, Eclipse windows)
├─ glEnable(GL_BLEND), glDisable(GL_CULL_FACE)
└─ RenderTargetAreas()                           — spawns 3D particles at task targets

├─ Early-out checks
├─ SetupRenderMatrices(ScreenOverlay)
├─ RenderMessages, RenderTitleBar, RenderTaskManager, etc.
├─ Standard cursor when task manager visible
├─ RestoreRenderMatrices
└─ glEnable(GL_CULL_FACE), glDisable(GL_BLEND)
```

`RenderTargetAreas` only creates particles via `ParticleSystem::CreateParticle` — it does
not draw geometry itself.  Verify this is safe to call before the particle system has
rendered (it is: particles are added to the list and rendered during `ParticleSystem::Render`
which already happened).  **Correction**: `RenderTargetAreas` runs *after*
`ParticleSystem::Render` currently, and the particles it creates would be rendered next
frame.  This ordering is preserved in the new layout since `Render3D` still runs after
`ParticleSystem::Render`.

#### `g_gameFont.BeginText2D()` removal

The current `Render()` wraps its 2D section in `g_gameFont.BeginText2D()` /
`g_gameFont.EndText2D()` — a *different* `TextRenderer` instance from the outer
`g_editorFont.BeginText2D()` used by `RenderFrame`.  Removing the inner pair is safe
because:
- Both `TextRenderer` instances compute identical ortho matrices (viewport-based).
- `RestoreRenderMatrices()` calls `SetupMatricesFor2D()`, not `g_gameFont.EndText2D()`,
  so it does not depend on `g_gameFont`’s saved matrix state.
- Text rendering uses whichever font instance calls `DrawText2D` — the active ortho
  matrix is the same regardless of which `TextRenderer` set it up.

- `taskmanager_interface.h` — Add `virtual void Render3D()`, `virtual void Render2D()` to base.
  (Only one derived class exists: `TaskManagerInterfaceIcons`.  Virtual is kept for
  consistency with the existing `virtual void Render() = 0` pattern.)
- `taskmanager_interface_icons.h` — Override both.
- `taskmanager_interface_icons.cpp` — Split `Render()`.

### 3. `UserInput` — trivial but state-sensitive

`UserInput::Render()` (user_input.cpp:109-158) is already pure 2D.  It calls
`BeginText2D`/`EndText2D` internally.

#### Change

Remove the internal `BeginText2D`/`EndText2D` pair.  The outer 2D pass provides the
ortho matrix.

**Critical**: lines 121–131 are **not** push/pop — they are hardcoded state changes:

```cpp
glEnable(GL_BLEND);           // line 121  ← OK for 2D pass
glDisable(GL_CULL_FACE);      // line 122  ← OK for 2D pass
glDisable(GL_DEPTH_TEST);     // line 123  ← OK for 2D pass
glDepthMask(false);           // line 124  ← OK for 2D pass
// ... EclRender() ...
glDepthMask(true);            // line 128  ← BREAKS 2D pass
glEnable(GL_DEPTH_TEST);      // line 129  ← BREAKS 2D pass
glEnable(GL_CULL_FACE);       // line 130  ← BREAKS 2D pass
glDisable(GL_BLEND);          // line 131  ← BREAKS 2D pass
```

After `EndText2D` is removed, lines 128–131 restore 3D-style state inside the 2D pass,
corrupting GL state for `TaskManagerInterface::Render2D()` and `GameCursor2D::Render()`
that follow.

**Solution**: Remove lines 121–131 entirely.  The outer 2D pass owns the GL state
contract: `GL_DEPTH_TEST` disabled, `glDepthMask(false)`, `GL_CULL_FACE` disabled,
`GL_BLEND` enabled.  `EclRender()` should not toggle these.  Add a comment at the
top of the 2D pass in `RenderFrame` documenting the state contract.

#### Files changed

- `user_input.cpp` — Remove lines 116 and 133 (`BeginText2D` / `EndText2D`).
  Remove lines 121–131 (state enable/disable block).  Add comment: `// Caller provides
  ortho matrix and 2D GL state (depth off, blend on, cull off).`

### 4. `Renderer::RenderFrame` — orchestrator

#### Before (current)

```cpp
void Renderer::RenderFrame(bool withFlip)
{
  SetOpenGLState();
  SetupMatricesFor3D();
  glClear(...);

  // 3D scene
  g_app->m_location->Render();          // or globalWorld
  g_explosionManager.Render();
  g_app->m_particleSystem->Render();

  // Mixed — each manages own matrices
  g_app->m_userInput->Render();
  g_app->m_gameCursor->Render();
  g_app->m_taskManagerInterface->Render();
  g_app->m_camera->Render();

  // 2D HUD
  g_editorFont.BeginText2D();
  // ... FPS, lag, logo, start sequence, Darwin logo ...
  g_editorFont.EndText2D();

  if (withFlip) g_windowManager->Flip();
}
```

#### After (proposed)

```cpp
void Renderer::RenderFrame(bool withFlip)
{
  SetOpenGLState();
  SetupMatricesFor3D();
  glClear(...);

  // ====== 3D PASS ======
  if (g_app->m_locationId != -1)
    g_app->m_location->Render();
  else
    g_app->m_globalWorld->Render();

  g_explosionManager.Render();
  g_app->m_particleSystem->Render();

  g_app->m_gameCursor->PrepareCursorState();
  g_app->m_gameCursor->Render3D();
  g_app->m_taskManagerInterface->Render3D();

  if (g_app->m_startSequence)
    g_app->m_startSequence->Render3D();   // 3D grid (only during start sequence)
```

### 5. `StartSequence` — mixed 2D/3D (previously misclassified)

`StartSequence::Render()` (startsequence.cpp:88-236) was listed as pure 2D in the
HUD block, but it is actually **mixed 2D/3D**:

- **Lines 89–170** (2D): Sets its own ortho projection via `OrthoOffCenterRH(0, 800, ...)`,
  disables depth test, renders screen-space quads (fade overlays) and text captions
  (`DrawText2D`, flashing cursor).
- **Line 170**: Calls `SetupMatricesFor3D()` — **switches to perspective**.
- **Lines 193–236** (3D): Renders a large-scale 3D grid (`glVertex3f` lines, fog, depth
  test enabled, modelview scale) behind the Darwinia title.  Only active when
  `timeNow > 50.0f`.

Currently this works because the 3D grid renders at the very end of `Render()` and no
further 2D drawing depends on the matrix state (the Darwin logo and `StartSequence`
never display simultaneously).  But placing it wholesale into the 2D pass would cause
the 3D grid to render with the wrong projection.

#### Proposed split

```
StartSequence::Render3D()                         — 3D grid behind title
├─ if (timeNow <= 50.0f) return              — early-out: grid not visible yet
├─ SetupMatricesFor3D (already active in 3D pass)
├─ Push modelview, Scale
├─ Fog, depth test, line rendering
├─ Pop modelview
└─ Restore fog to location fog

StartSequence::Render2D()                         — captions and fade overlays
├─ Set own ortho projection (OrthoOffCenterRH)
├─ glDisable(GL_DEPTH_TEST)
├─ Fade-in / fade-out quads
├─ Caption text with flashing cursor
└─ (does NOT call SetupMatricesFor3D — removed)
```

Note: `Render2D()` sets its own ortho projection (`OrthoOffCenterRH(0, 800, ...)`) that
differs from the `BeginText2D` ortho (viewport-based with pixel offsets).  This is
intentional — the start sequence uses a fixed 800-wide virtual coordinate system for
its captions.  This overwrite is harmless because `RenderHUD` restores the standard
ortho via subsequent `g_editorFont.DrawText2D` calls (which use `g_editorFont`'s own
coordinate mapping), and the Darwin logo block uses screen-pixel coordinates that work
with either ortho.

#### Files changed

- `startsequence.h` — Add `void Render3D()`, `void Render2D()`.  Keep legacy `Render()`
  during transition.
- `startsequence.cpp` — Split `Render()` at line 170.  `Render2D()` gets lines 89–170
  (minus the `SetupMatricesFor3D` call).  `Render3D()` gets lines 170–236.
- `renderer.cpp` — Add `StartSequence::Render3D()` call in 3D pass (conditional on
  `g_app->m_startSequence != nullptr`).  `RenderHUD()` calls `Render2D()` instead of
  `Render()`.

| Risk | Impact | Mitigation |
|---|---|---|
| **2D cursors drawn after 3D markers changes visual layering** | Cursors may appear on top of 3D elements they previously interleaved with | This is actually *correct* — screen-space cursors should always be on top. Verify with gameplay. |
| **`RenderTargetAreas` spawns particles that won't render until next frame** | Already the case today — no change. | Document in code comment. |
| **External callers of `RenderStandardCursor`** | `TaskManagerInterfaceIcons::Render` calls it directly (line 416) | Moves to `GameCursor2D` which owns `m_cursorStandard`. Update call site in Step 5. |
| **Eclipse windows (`EclRender`) need ortho matrix** | `UserInput::Render` currently provides this via `BeginText2D` | Outer 2D pass now provides it. Verify Eclipse mouse coordinates are correct. |
| **`GameCursor::PrepareCursorState` queries gameplay state** | If called at a different point in the frame, selection/highlight results could differ subtly | `PrepareCursorState` is called at the same relative position (after particles, before cursor drawing). | **Legacy `Render()` wrapper on `GameCursor`** | Minor API surface increase during transition | Remove in Step 7 once `RenderFrame` is rewired. |
| **Shared `MouseCursor` lifetime** | `GameCursor2D` holds references to `m_cursorPlacement`/`m_cursorDisabled` owned by `GameCursor` | `GameCursor` must outlive `GameCursor2D`. Enforce in `GameApp` init/shutdown order. |

| **`StartSequence::Render` is mixed 2D/3D** | Plan originally classified it as pure 2D. The 3D grid (lines 170–236) would render with wrong projection inside the 2D pass. | Split into `Render3D()`/`Render2D()` in Step 1. |

## Consequences

### Positive

- **Fewer PSO switches**: all 3D draws share depth-on PSOs; all 2D draws share
  depth-off PSOs.  Eliminates 6+ unnecessary pipeline state changes per frame in
  `GameCursor` alone.
- **Single ortho transition**: one `BeginText2D` / `EndText2D` pair instead of 4+.
- **Upload buffer locality**: `DrawConstants` uploads cluster by projection type,
  improving cache coherence in the ring buffer.
- **SpriteBatch ready**: `GameCursor2D` is an isolated class whose only rendering calls
  are 2D `glBegin`/`glEnd` quads — a 1:1 replacement target for `SpriteBatch::Draw()`.
  The 3D path in `GameCursor` is untouched when that migration happens.
- **Clearer ownership**: 3D cursor bugs → `gamecursor.cpp`, 2D cursor bugs →
  `gamecursor_2d.cpp`.  Each file has a single rendering paradigm.
- **Correct layering**: 2D overlays are guaranteed to draw on top of all 3D content.

### Negative

- **More files**: two new files (`gamecursor_2d.h/.cpp`), `StartSequence` split, and a
  new `GameApp` member.
- **Behaviour-sensitive reorder**: if any subsystem's 2D drawing depends on 3D state
  written by a *different* subsystem's `Render()` call (e.g. reading back a depth value),
  the reorder will break it.  Audit required — no such dependency found in the current
  code.
- **Pre-computed screen projections**: `RenderSelectionArrows` can no longer call
  `Get2DScreenPos` inline because the 2D pass has overwritten the projection matrices.
  Screen positions must be pre-computed in `PrepareCursorState` during the 3D pass.
  This adds complexity to the frame state but makes `GameCursor2D::Render` a pure
  2D draw call with no hidden matrix dependencies.
- **Testing overhead**: each migration step needs visual verification across global world,
  location gameplay, task manager open/closed, and cutscene/non-interactive camera modes.
- **Shared cursor indirection**: `GameCursor2D` accesses `m_cursorPlacement` /
  `m_cursorDisabled` via `GameCursor&` — one extra pointer hop per use.  Negligible at
  cursor-draw frequency.
- **`selectionArrowBoost` write-back**: The decrement in `RenderSelectionArrows` must be
  written back to `GameCursor` after `GameCursor2D::Render()`.  Minor complexity cost.
